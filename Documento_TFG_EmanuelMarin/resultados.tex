\subsubsection{Arduino UNO R3, motor BYJ-48 y controlador ULN2003}

Para el control del motor (utilizando el controlador) se empleó la biblioteca \textit{Stepper} (importada a través de su \textit{header} \textit{Stepper.h}) para hacer girar el motor.

Primeramente se procedió a calcular la cantidad de pasos para hacer rotar el motor la cantidad necesaria (90 grados). 
De la hoja de datos del motor se puede extraer que un giro completo de 360 grados tendrá 4096 pasos. Por lo que para obtener un giro de 90 grados se utilizan 1024 pasos por giro. 
Se utiliza la función \textit{setUp} (se ejecuta una única vez antes de la ejecución del resto del código) para realizar la conección con el puerto serial y configurar la velocidad en pasos por segundo del motor.

A continuación se muestra el código de que se almacena en la memoria \textit{Flash} del Arduino para que se cargue y ejecute luego del arranque.

El código permite que se espere a que el mensaje llegue para realizar un giro. Una vez este se complete se devuelve el mensaje a la computadora para que se tome la fotografía.
Además, durante el proceso de desarrollo se percibió que el Arduino mantenía los pines encendidos y terminaba por calentar el motor y su controlador, por esta razón se añadieron 4 lineas de código en donde se apaga cada pin utilizado para evitar este calentamiento.

También se detectó en casos muy poco comunes que el mensaje se corrompió por razones desconocidas por lo que se incluyó un mensaje de respuesta especial para cuando el mensaje que llega no es el que debería.

\begin{lstlisting}

#include <Stepper.h>
#define G_RATIO 64 //gear ratio

Stepper stepper (G_RATIO, 8, 10, 9, 11); //Set pins

String msg = ""; //empty string for message loading


void setup() {

  Serial.begin(9600); //using port 9600
  stepper.setSpeed(1000); //set speed for optimal time 
  
}

void loop() {
  while (!Serial.available()) {} 
  
  while (Serial.available())
  {
    if (Serial.available()>0){
    
      msg = Serial.readString();
      if (msg == "1"){
        stepper.step(1024);
        Serial.println(msg); //returns the message so PC knows the turning ended
        //Procede to turn off all pins to avoid overheating
        digitalWrite(8,LOW);
        digitalWrite(9,LOW);
        digitalWrite(10,LOW);
        digitalWrite(11,LOW);  
        }
      else{
        msg = '-1'
        Serial.println(msg); //for debugging in case msg gets corrupted
        }
    }
  }
}
\end{lstlisting}

Además, durante el proceso de pruebas se notó que la iluminación del cuarto podía afectar la forma en la que la cámara tomaba la foto, o sea se podían percibir tonalidades de amarillo debido a la fuente de luz de la lampara, pero esta debería ser blanca. Para evitar este fenómeno se implementó un corrector de balance de blanco. El cual corrige este efecto de la imagen.

Este filtro consiste en una operación sobre la imagen convertida al modelo de color \textbf{LAB} (L - \textit{ligthness}, A - balance entre rojo y verde, B - balance entre azul y amarillo). Se parte de que el color promedio reflejado visto en la fotografía es el de la luz, esto nos permite saber cuanto se debe mover el balance de blanco de los colores para obtener la corrección. A esto se le conoce cómo la \textit{grayworld assumption} (suposición del mundo gris). \cite{color_correct}

La ecuación consiste en:

\begin{equation}
    Xn = X - (Xa - 128) * L/255.0 * Cf 
\end{equation}

Donde \textbf{X} es el valor de \textbf{A} y \textbf{B} para cada píxel (la corrección se hace sobre cada valor de \textbf{A} y de \textbf{B}), \textbf{Xn} es el valor resultante nuevo para el valor de \textbf{A} y \textbf{B}, \textbf{Xa} es el promedio de todos los valores para \textbf{A} y \textbf{B} de toda la imagen, y \textbf{Cf} es el factor de corrección (este puede cambiar dependiendo los resultados). Después de probar diferentes valores se llegó a que un \textbf{Cf} de \textbf{1.1} producía los mejores resultados.

\subsubsection{Motor de decisión - Modelo de red neuronal convolucional}

Se partió del modelo propuesto en el capitulo anterior. Figura \ref{fig:initial_model}.
Cabe resaltar que debido a las limitaciones de potencia de la computadora de trabajo los entrenamientos duraron entre 4 y 15 horas. Estos dependiendo de la cantidad de datos, parámetros, tamaño de la red, entre otros.


Este modelo luego fue alimentado utilizando las unidades iniciales brindadas al principio del provecto. Correspondían a 60 imágenes en total, 50 de aprendizaje y 10 de validación.
Para comparar los resultados se utilizaron \textit{logs} a terminal, matplotlib y TensorBoard. Esto pues es importante comparar los resultados y combinaciones que los brindan para poder elegir un modelo efectivo.

Sin embargo, esta cantidad de datos arrojó resultados pésimos sin importar cuantas épocas de entrenamiento se hicieran, pues la precisión de validación consistía en que siempre se decidía una misma categoría. Ya que el 70\% de las unidades que se le alimentaron eran ''buenas''. A continuación se muestran los resultados obtenidos para 20 épocas:

Para este modelos se cambiaron los parámetros:
\begin{itemize}
    \item Inicializador de Kernel: Se probaron Normal y Uniforme.
    \item Regularización: Se utilizó L1 y L2 de 0.001, 0.01 y 0.1.
    \item Activación: Se probaron diferentes activaciones: \textit{ReLu}, \textit{TanH}, \textit{Sigmoid} y \textit{LeakyReLu}.
    \item Tamaño de lote: Se probaron con lotes de 1, 5, 10 y 25 imágenes.
    \item Normalización de datos: división por 255.0, división por desviación estándar para muestra y resta de media para muestra.
    \item Tamaños de imagen: 100, 150, 300 y 500 píxeles de alto y ancho
    \item \textit{Pooling}: Se probó la red con y sin capas de \textit{pooling}.
    \item Normalización por lote: Se probó la red con y sin normalización por lote después de las capas convolucionales.
\end{itemize}

Se realizaron muchas combinaciones de estos parámetros, sin embargo no se logró obtener una mejora en la precisión o la perdida. Por lo que se concluyó que el problema estaba en la cantidad de datos y/o en el hecho de que fueron tomadas en condiciones no estandarizadas.

Se probó con un paquete de datos llamado \textit{horses\_and\_humans}, este provee imágenes generadas por computadora de caballos y seres humanos para clasificar. Se probó con este paquete de datos para comprobar que no se estuvieran cometiendo errores a la hora de crear la red, pues si es capaz de aprender hasta cierto grado de precisión en este problema es poco probable que el no aprender el otro problema esté relacionado a algún problema en la red (gradientes perdidas, pesos sobre escritos, neuronas muertas, etc). \begin{equation}
    Precision = \frac{192}{192 + 50} = 0.793
\end{equation}
\begin{equation}
    Recall = \frac{192}{192 + 8} = 0.96
\end{equation}
\begin{equation}
    Puntaje f = 2 * \frac{0.793 * 0.96}{0.793 + 0.96} = 0.868
\end{equation}

Dado que estos resultados no resultan deseables según los valores buscados se procede a modificar la red. Esta vez dado que se tienen buenos resultados y se probaron bastantes combinaciones de parámetros se procede a cambiar el tamaño de la red, buscando una mayor capacidad de representación y una posible mejora en el aprendizaje de las características de las imágenes.

Se cambió:

\begin{itemize}
    \item Tamaño de capas: Capas convolucionales de 32, 64 y 128.
    \item Cantidad de capas: Se incrementó la cantidad de capas convolucionales y de \textit{pooling} en 1, 2 y 3.
    \item Se cambió la cantidad de neuronas en la capa densa después de la capa de \textit{flatten} en el doble y el triple.
\end{itemize}

La combinación que dio los mejores resultados fue:

\begin{itemize}
    \item Parámetros iguales al modelo anterior.
    \item Cantidad de capas: Se incrementó la cantidad de capas convolucionales y de \textit{pooling} en 2
    \item Cantidad de neuronas: Las capas convolucionales ahora son de: 32, 64, 128, 256, 256.
    \item La capa densa ahora es de 1024 neuronas.
    \item Se realizó aprendizaje durante 100 épocas.
\end{itemize}

\subsection{Diseño del prototipo - \textit{hardware}}

El prototipo planteado según los resultados obtenidos muestra que cumple aquello deseado, este es capaz de sostener las unidades de \textit{balloon} sin dañarlos, los permite rotar para tomar las capturas, y permite tomar las fotografías. Esto gracias al uso de una base impresa dentro de la misma empresa, pues así se garantiza que se cumplan los estatutos y regulaciones internas, además el costo es reducido.

El uso del Arduino es una excelente decisión para garantizar la transparencia pues al ser una plataforma la cual se puede comprobar con facilidad los componentes que contiene es posible desarrollar una versión compacta (sin componentes extra). Incluso se puede diseñar e implementar un circuito integrado que realice sus funciones. Otra ventaja de este acercamiento es que permite expandir el alcance del proyecto a futuro, pues es posible hacer uso de los recursos restantes del Arduino para otras funciones.

Durante las pruebas realizadas fue posible observar que en ninguna ocasión se dio la toma de alguna fotografía mientras la unidad se encontraba girando. Esto comprueba que la comunicación en ambas direcciones para confirmar una acción fue un acierto de diseño.

El \textit{hardware} a pesar de su apariencia un poco ''rustica'' o tosca cumple con su propósito, además de los objetivos del proyecto, de ser ''transparente'' esto pues permite a futuros desarrollos percibir la totalidad de sus componentes y garantizar una rápida replicación y modificación a cómo se precise. Un aspecto muy importante, ya mencionado en la sección de validación del modelo en el capitulo de metodología, es que por la naturaleza de la aplicación (industria médica) se deseaba un limite muy alto de \textit{recall}, pues al tomar las unidades defectuosas cómo nuestros \textit{positivos} los falsos negativos lo reducirán, ya que un falso positivo causaría perdida monetaria pero un falso negativo puede significar la muerte de un paciente y aunque la perdida monetaria no es deseada es indiscutiblemente preferible que la perdida de una vida. Al comienzo del desarrollo de este proyecto los requerimientos fueron establecidos para asegurar que se siguiera este principio, el de proteger la vida a cualquier costo, e incumplir con este requerimiento era, instantáneamente, considerado cómo que el concepto no funciona y debe ser desechado; el haber logrado cumplir con este requerimientos e incluso superar el limite es considerado uno de los mayores logros de este proyecto, pues demuestra que es posible automatizar más una industria tan delicada manteniendo altos estándares de seguridad y confiabilidad.


Con respecto a la duración del análisis, el proceso actual toma al rededor de 2.5 minutos para comprobar 3 unidades. Por lo que una duración de 0.20 segundos es por mucho superior a esta duración, tomando un total de 0.6 segundos para realizar el análisis de 3 unidades, a esto se le añade que el método es no destructivo y no requiere tiempo de entrenamiento al personal.
Los únicos puntos de consumo de tiempo se encuentran en la colocación en posición de la unidad y de la cámara, esta última sólo requiere ser colocada una única vez al comienzo de las pruebas. El tiempo para la colocación de la unidad fue medido durante las pruebas: se encontró que consiste en \textbf{30} segundos inicialmente y luego se registra una mejora (al acostumbrarse al proceso) que permite alcanzar tiempos de \textbf{15} a \textbf{20} segundos, de estos datos se puede extraer que el tiempo de duración para 3 unidades de 60.6 a 90.6 segundos aproximadamente , o sea se tarda entre un 40\% a un 60\% del tiempo original; esto aunque no fue contemplado en los requerimientos pues no era relevante para la validación del concepto resulta ser un punto muy a favor de la solución implementada.

Se puede extraer, también, que el incremento de neuronas y capas en el modelo resultó efectivo. Esto incrementó la capacidad de representación de la red y llevó a resultados más efectivos. Una vez obtenidos los resultados finales, se puede confirmar que la metodología de desarrollo fue efectiva pues se obtuvieron buenos resultados, y se cumplieron con los requerimientos planteados.

Cabe señalar, que de las configuraciones probadas se obtuvieron resultados muy interesantes. Para esta solución no fue necesario implementar una arquitectura más compleja que una red neuronal convolucional convencional, su tamaño ni siquiera puede ser considerado demasiado grande. Además, no fue necesario hace uso de los parámetros utilizados comúnmente para evitar los grandes problemas al utilizar las redes neuronales en aplicaciones muy complejas cómo sería usar regularización o normalización de lote para evitar un sobre ajuste, incluso al haber sido utilizadas no se obtuvieron mejoras o incluso los resultados fueron inferiores. Incluso es posible notar que los mejores resultados se obtuvieron con modelos relativamente simples y que no utilizaron los aspectos más complicados vistos en las redes neuronales de la industria ''hoy por hoy''. Esto demuestra que no siempre lo más complejo o ''poderoso'' es lo mejor y que soluciones simples son capaces de alcanzar resultados excepcionales contra implementaciones más complejas. 

Otro aspecto a resaltar es que incluso no fue necesario utilizar tamaños de imágenes demasiado grandes, un tamaño de 300x300 dio mejores resultados que el tamaño de 500x500, permitiendo un entrenamiento más rápido y menor consumo de memoria sin perdida de rendimiento de la red neuronal, esto es muy interesante dado que la cantidad de información es bastante menor; para el tamaño de 300x300 con RGB se tienen: 270000 valores (300x300x3) y para el de 500x500 se tienen: 750000, o sea se obtuvieron resultados buenos utilizando un 74\% menos valores (estos en punto flotante precisión simple). Un comportamiento similar, en donde aunque se utilizan menos datos por imagen los resultados no hayan empeorado, es con el caso de las capas de \textit{pooling} que reducen considerablemente la dimensionalidad de los datos entre capas convolucionales.

En resumen, de los resultados se puede extraer que la metodología propuesta probó ser efectiva para lograr una solución deseable para el problema, se logró cumplir con los objetivos e incluso se lograron mejoras en aspectos que no fueron contemplados cómo requerimientos.
