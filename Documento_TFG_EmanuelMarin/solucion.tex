\section{Propuesta de diseño del motor de decisión - red neuronal convolucional}

El motor de decisión no es más que el algoritmo que se encargará de analizar la fotografía y determinar si esta corresponde a una unidad que pasa o que no pasa los requerimientos.

Debido a los resultados obtenidos por la investigación realizada antes de la concepción de este proyecto se concluyó que no se debía realizar un método matemático explicito pues se ''topó con pared''.
Por esta razón se tomó la decisión de desarrollar un modelo de aprendizaje de maquina que permitiera de forma no explicita inferir la relación entre las características presentes en las fotografías de las unidades para poder decidir si las unidades cumplen, o no, con los requerimientos de presión soportable.

Por lo antes mencionado se optó por desarrollar una red neuronal convolucional. Gracias a su funcionamiento es posible para esta aprender características significativas de las imágenes que permiten clasificarlas en las categorías deseadas. En nuestro caso queremos poder clasificar las imágenes en dos categorías: pasa o falla. Esto referente a si pasa el mínimo de de presión que debería soportar antes de reventar, o si falla esta prueba.

Ahora, debido a la naturaleza de las redes neuronales no es determinista decir que un modelo que sirve para una aplicación será el mejor para otra o siquiera servirá para esta. Además, los diferentes parámetros a utilizar para la red (función de activación, tamaño de kernel, cantidad de capas, cantidad de neuronas, función de perdida, normalización de datos, cantidad de épocas, tamaño de lote y tamaño de muestra, entre otros) no necesariamente serán los mejores para una aplicación simplemente porque funcionaron para otra. \cite{khanbook}

Por esto se propone trabajar bajo un método iterativo de prueba. Donde se obtengan resultados, se analicen y se realizan cambios en el modelo para intentar obtener mejores resultados, esto hasta que se obtengan resultados satisfactorios. Por supuesto se debe partir de un modelo básico que tenga sentido para la aplicación que se busca.

Se propone comenzar con el siguiente modelo (basado en un ejemplo básico provisto por la documentación de TensorFlow), pues este es utilizado para obtener un buen resultado en la clasificación de animales que aunque no se adapte a nuestro problema (y por lo antes mencionado) muestra que tiene una buena capacidad de representación de problemas, por lo que es considerado un buen lugar para comenzar:

\begin{itemize}
    \item Modelo secuencial.
    \item Datos en RGB de 300x300, con cada píxel multiplicado por un factor de 1/255 para obtener valores entre 0 y 1. Se propone trabajar en RGB y no en escala de grises para no perder ningún tipo de información por más mínima que pueda ser.
    \item Tres capas convolucionales de dos dimensiones, con unidad lineal rectificada de activación.
    \item kernel de tamaño 3x3 en todas las capas convolucionales.
    \item Tres capas de \textit{pooling} de máximos, cada una después de una capa convolucional.
    \item La primer capa convolucional de 16 neuronas, las dos siguientes de 32 y 64 respectivamente.
    \item Luego de las capas convolucionales y de \textit{pooling} una capa de aplanado (flatten).
    \item Antes de la capa de salida una capa densa (densamente conectada) de 512 neuronas, activación \textit{ReLu}.
    \item Cómo capa de salida una capa densa de una neurona, activación \textit{softmax}. Esto pues el \textit{softmax} produce salidas entre 0 y 1, siendo 0 una categoría y 1 la otra. Entre más cercano de uno u otro así de ''segura'' estará la red de la categoría.
    \item Cómo optimizador una función \textit{Adam} con un \textit{learning rate} de 0.001 (para evitar cambios muy grandes en los valores que puedan ser contraproducentes).
    \item Y para calcular la perdida: una función de entropía cruzada binaria. Esto pues esta es especial para problemas de clasificación binaria.
    \item Modelo sin regularización.
\end{itemize}

\subsection{Proceso de aprendizaje de la red neuronal convolucional}

El proceso de aprendizaje será estandarizado bajo los siguientes pasos:

\begin{itemize}
    \item Entrenar el modelo actual con los datos de entrenamiento.
    \item Probar el modelo con los datos de validación.
    \item Utilizar un salvador de estados, ara que se almacene el modelo con los mejores valores de perdida en validación. Así si el mejor modelo es obtenido antes de que se completen la totalidad de las épocas este sea almacenado y no se vea sobre escrito por los obtenidos en épocas posteriores que serán inferiores en rendimiento.
    \item Obtener una matriz de confusión con la distribución de predicciones para cada categoría.
    \item Calcular los valores para precisión, \textit{recall} y \textit{f-score}.
    \item Si los resultados para estos valores no son deseables según lo que se estipula más adelante en la sección de validación el modelo debe ser sometido a modificaciones para repetir este proceso.
    \item Una vez obtenidos resultados deseables el modelo puede ser sometido a otras modificaciones con la intención de mejorar los resultados.
\end{itemize}

Para los cambios al modelo se contemplan los siguientes parámetros y características:

\begin{itemize}
    \item Épocas: Consiste en la cantidad de iteraciones que se realizaran sobre el conjunto de datos de aprendizaje. Se propone utilizar: 10, 30, 50, 80 y 100 épocas.
    \item Inicializador de Kernel: Este consiste en los valores iniciales que tendrán los kernel contenidos en el modelo. Se propone probar con inicializadores con distribución normal y distribución uniforme.
    \item Regularización: La regularización consiste en un valor de penalización extra en el calculo del error, que permite evitar que el modelo se sobre ajuste a la población de entrenamiento. Se propone utilizar valores cómo: 0.1, 0.01 y 0.001, esto en el caso de que se obtenga sobre ajuste, para ambos tipos: L1 y L2.
    \item Activación: Se propone utilizar las funciones de activación más comunes: \textit{ReLu}, \textit{TanH}, \textit{Sigmoid} y \textit{LeakyReLu}. Comenzando por \textit{ReLu} y en caso de que los resultados no sean deseables probar con otras. En caso de los resultados no muestren cambios entre una y otra, se propone elegir cualquiera arbitrariamente.
    \item Tamaño de lote: Consiste en la cantidad de imágenes que se alimentan a la red ''al mismo tiempo''. Se proponen lotes a conveniencia, el valor observado más comúnmente es de 32 pero se propone probar con valores en el rango de 1 a 50 imágenes.
    \item Normalización de datos: se propone utilizar normalización por división por 255.0 (todos los valores de píxel entre 0 y 1), división por desviación estándar para muestra y resta de media para muestra.
    \item Tamaños de imagen: Se propone emplear tamaños de 100, 150, 300 y 500 píxeles de alto y ancho. Esto para probar tamaños que permitan pruebas más rápidas sin que se pierda la información relevante en la reducción de la escala.
    \item \textit{Pooling}: Esta consiste en una capa que permite reducir la dimensionalidad de los datos manteniendo la relación entre estos. Se propone utilizar \textit{pooling} basado en la función \textit{max} (valor máximo), conocida cómo \textit{pooling}.
    \item Normalización por lote: En casos donde se cuenta con sobre ajuste es posible utilizar normalización por lote para reducirlo. Esta consiste en una capa que normaliza las salidas de la capa anterior antes de pasarlas a la siguiente capa.
\end{itemize}

El objetivo de cambiar estas características y parámetros es buscar un modelo que se adapte mejor al problema y que los resultados sean más deseables. Debido a la gran cantidad de combinaciones posibles se propone utilizar \textit{early stopping} o \textit{parada temprana}, la cual consiste en que al pasar cierta cantidad de épocas y no verse un cambio (mejora) en una métrica establecida (en este caso perdida de validación) con una precisión establecida (valor arbitrario elegido por el programador) se interrumpe el proceso de aprendizaje. Se propone utilizar una espera de 10 épocas con una precisión al tercer valor decimal (0,001).

Cómo un tipo de apéndice a este proceso se propone también que en caso de que se dude de la ''correctitud'' del modelo, o sea, se sospecha que existen errores en su código cómo que se estén perdiendo valores de pesos o siendo sobrescritos, etc, se realicen pruebas sobre un paquete de datos ya probado y que corresponda a un problema complejo. En este caso se propone el paquete de datos llamado: \textit{horses\_and\_humans}.

Para el entrenamiento se tiene un grupo de imágenes inicial provistas por la empresa, sin embargo este paquete de datos corresponde a 60 imágenes en total, por lo que al dividirlo en paquete de entrenamiento y validación se tendrían muy pocas imágenes por lo que se sospecha que no se obtendrán buenos resultados, considerando que una solución por redes neuronales convolucionales suele requerir un alto número de datos de entrenamiento y una cantidad de imágenes de validación que sea representativa de las posibles entradas obtenidas durante una ejecución final del proyecto. Se propone separar el paquete inicial en 50 imágenes de entrenamiento y 10 de validación, estas siendo: 40 unidades que cumplen y 10 que no en entrenamiento; 7 que cumplen y 3 que no en validación, pues esta es la forma más efectiva que se encontró para su división haciendo bastante clara la necesidad de más datos de entrenamiento. 

Por ende, se propone planear un proceso de extracción de datos, el cual permita obtener una cantidad considerable de datos de entrenamiento y validación, esta realizada por medio de extraer unidades del proceso real de producción, ser fotografiadas bajo la luz polarizada y pasar a un proceso de ''ruptura'' en el cual se revientan las unidades y se determina el valor exacto de presión al que se provocó la ruptura y se determina si la unidad cumplía o no. Cabe mencionar que debido a la efectividad del proceso se espera que sea complicado obtener muchas unidades defectuosas, por lo que se propone coordinar una ejecución de producción bajo variables que provoquen que las unidades no cumplan, para esto se utilizaran documentos internos existentes donde se especifican investigaciones que determinaron cuales variables (y sus valores) suelen producir unidades que no cumplen los requerimientos.

Debido a la naturaleza de este proyecto (prototipo) la cantidad de unidades que se pueden pedir es limitada (por razones de capacidad y demanda de producción), este valor corresponde a 400 unidades en total, para un total de 1600 fotografías, además a este proceso se le agrega que otros factores pueden provocar que las unidades no funcionen para el objetivo del proyecto por defectos de otras índoles que descartan la unidades antes de esta etapa de revisión; por lo que la cantidad de datos puede ser menos a 1600 fotografías.

Es importante estandarizar las condiciones en las que se toman los datos y que además estas sean representativas de las condiciones en las que se llevará a cabo el uso final del prototipo, por lo que se debe realizar la captura de las fotografías con el prototipo final de la misma forma en la que se tomarían las fotografías de las unidades en una ejecución de prueba o final (producción real).

A continuación en la sección de validación del modelo se propone una distribución porcentual de la cantidad de datos para entrenar y validar, esta buscando que se cumpla lo mencionado: gran cantidad de datos de entrenamiento y validación que además sean descriptivos del problema).


\subsection{Validación del Modelo de \textit{Deep Learning}  (Red neuronal convolucional)}

La validación del modelo es una de las partes más importantes pues es con lo que se prueba que el modelo diseñado funciona para el problema.

Para validar el modelo se propone:

\begin{itemize}
    \item Comenzar probando el rendimiento del modelo en paquetes de datos para problemas ''simples''. El objetivo de esto es comprobar que el modelo es capaz de ''aprender'' problemas más simples y, sobre todo, comprobar que el modelo aprende. Esto permite realizar \textit{debugging} del código del modelo.
    \item Una vez se compruebe que el modelo no posee errores en su código se debe proceder a alimentar los datos de entrenamiento preliminares (paquete de datos brindado al comienzo del proyecto por la empresa), este paquete es pequeño pero sirve como previa del problema.
    \item Una vez obtenidos los datos de entrenamiento y validación finales estos deben ser alimentados al modelo para comprobar su efectividad en el problema y realizar el proceso iterativo mencionado anteriormente.
    \item Se propone utilizar un 10\%, como mínimo, del total de los datos  para crear el paquete de datos de validación. Estos deben ser representativos de los 3 grupos principales en los que se pueden dividir las unidades (muy buenas, muy malas y valores cercanos al limite entre las categorías).
    \item Se propone utilizar \textit{TensorBoard} y/ola biblioteca \textit{MatPlotLib} para visualizar los resultados del modelo de forma gráfica.
    \item Se espera que este prototipo (al funcionar cómo prueba de concepto) tenga una precisión no menor al \textbf{85\%} en validación, el \textit{recall} no debe ser inferior a un \textbf{90\%} y el \textit{f-score} no puede ser inferior a \textbf{87.4\%}, para que sea aceptable y se pueda considerar el concepto como probado. Debido a la naturaleza delicada del negocio (médico) se toma positivo cómo las unidades que fallan y las unidades que pasan cómo negativos. Esto para que el \textit{recall} refleje la delicadeza de una unidad que falla pasando cómo buena.
    
\end{itemize}
